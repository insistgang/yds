# LinkAble Agents Guide

> 版本：V2.2（对齐 PRD v2.2）
> 当前方向：基于边缘智能的无障碍辅助通行原型系统
> 最高约束：P0 优先、数据优先、稳定演示优先

---

## 1. 文档定位

本文件用于指导 LinkAble 项目的研发协作 agent、AI coding agent 和工程执行者，在需求拆解、代码实现、文档评审、联调演示和提交材料准备中保持一致。

**它不是终端用户说明书，也不是论文正文，而是把 PRD v2.2 中确定的新方向转成可执行协作规范。**

所有 agent 默认遵守：

```text
不要把 LinkAble 做成泛化 AI 助手。
不要把 LinkAble 做成完整导航 App。
不要把 LinkAble 包装成已经成熟量产的穿戴设备。
当前只做可稳定演示的无障碍辅助通行原型。
```

## 2. 项目新定位（PRD v2.2）

LinkAble 当前版本是一套 **基于边缘智能的无障碍辅助通行原型系统**。

它服务的核心问题是：

- 盲道是否被占用
- 前方是否有台阶
- 坡道或可通行入口在哪里
- 路面是否有临时障碍
- 用户能否听到简短、明确、可执行的语音提示

**系统主链路：**

```text
摄像头/图片/视频输入
  -> YOLO 目标检测
  -> 事件聚合与去抖
  -> 模板化中文提示
  -> presenter_female 本地语音播报
  -> 结构化事件记录
  -> [P1] 云端大模型长文本报告生成（Kimi/Qwen）
```

**一句话记忆：**

```text
无人驾驶让车看懂路，LinkAble 让弱势行人听懂路、让管理者看见盲区。
```

## 3. 核心原则

### 3.1 P0 是唯一主线

P0 必做：

- `blind_road_occupied` 盲道占用检测
- `stairs` 台阶识别
- `ramp` 坡道识别
- `road_obstacle` 路面障碍检测
- 连续帧事件聚合与去抖
- 模板化中文提示（<50字短句）
- 本地音频播报（presenter_female）
- 结构化事件记录
- 校园最后 20 米演示路线（A-D 点位）

P1 只在 P0 稳定后推进：

- 检测结果上报（FastAPI）
- 云端大模型语义增强（长文本报告）
- Flutter 展示页
- 红绿灯/人行横道识别

P2 本期不做：

- 实时大模型对话增强（避免延迟）
- 用户主动语音查询与动态路线规划
- 志愿者 LBS 互助
- 设施众包评分地图
- WebSocket 实时推送
- 多用户并发平台
- 商店上架
- 量产级外壳和工业设计

### 3.2 部署形态不要绑死

当前 PRD 允许三种演示形态：

| 形态 | 当前定位 | 使用场景 |
|---|---|---|
| **固定点位监测** | **推荐主形态** | 盲道入口、坡道入口、楼宇门口 |
| **手持/推车式巡检** | **推荐辅助形态** | 校园、社区、地铁口无障碍环境巡检 |
| 短距离胸前/肩带式佩戴 | **仅展示形态** | 拍摄演示视频片段，不作为现场主演示 |

任何 agent 不得把项目重新收窄成"必须长期穿戴 Jetson 到处跑"。

### 3.3 本地优先，云端增强

- 能本地播报就不要依赖云端
- 能用缓存 mp3 就不要现场请求 TTS API
- 能用模板解决就不要强依赖 LLM
- 能用视频/图片序列演示就不要被 CSI 相机问题完全卡死
- **云端大模型只做事后报告增强，不做实时推理**

### 3.4 隐私是硬约束

- 不存储原始视频
- 不上传连续图像流
- 可选上报只传结构化事件
- 位置数据默认匿名化
- 提交材料不得出现导师、学校等匿名违规信息
- 演示视频着装要求：正装或商务休闲+深色长裤/长裙

### 3.5 数据价值优先（PRD v2.2 核心主张）

**这是项目最有可能真正落地的价值。**

- **核心目标**：让无障碍环境的隐性问题变可见。盲道占用、坡道杂物、路面临时障碍——这些在城市管理者视角是隐性的，缺乏系统化数据支撑。
- **输出形态**：结构化事件记录（时间、位置、类型、置信度），转化为可量化、可追溯、可治理的数据。
- **服务对象优先级**：P0 第一位用户是"校园/社区无障碍管理方"，通过数据支撑治理决策；第二位才是视障用户的语音辅助。
- **论文/答辩必讲**：不要只讲"我们做了一个语音提醒设备"，要讲"我们提出了一种从机动车道感知到人行无障碍空间感知的场景迁移方法，并通过结构化记录支撑城市治理"。
- **禁止**：将项目窄化为"导盲设备"、弱化数据治理价值。

## 4. 统一产品口径

所有 agent 对外统一这样描述：

> LinkAble 是一套基于边缘智能的无障碍辅助通行原型系统。它借鉴自动驾驶中的环境感知与边缘推理思想，但把感知对象从机动车道迁移到人行无障碍空间，把输出从车辆控制转化为面向弱势行人（视障及行动不便人群）的语音辅助提示，并通过结构化事件记录支撑城市无障碍设施的长效治理。

**一句话记忆：**

```text
无人驾驶让车看懂路，LinkAble 让弱势行人听懂路、让管理者看见盲区。
```

**禁止口径：**

- "我们已经做出了成熟导盲设备"
- "可以替代盲杖或导盲犬"
- "可以保证视障用户完全安全"
- "我们的核心是 App/云平台/志愿者系统"
- "系统支持实时大模型对话"

**竞品对比口径（论文/答辩/展架可用）：**

| 方案 | 形态 | 感知方式 | 提示方式 | 成本 | 局限 |
|---|---|---|---|---|---|
| 传统盲杖 | 手持 | <1m 物理探测 | 触觉 | 低 | 无法识别远距离障碍 |
| 导盲犬 | 动物 | 视觉+训练经验 | 牵引+行为 | 20万+ | 全国仅400只 |
| 导航 App | 手机 | 路径规划 | 语音导航 | 免费 | 无局部环境感知 |
| 智能盲杖（WeWALK） | 手持/挂载设备 | 超声波+激光 | 震动+声音 | 千元级 | 无法识别障碍类别 |
| **LinkAble** | **边缘节点** | **YOLO 视觉识别** | **模板化语音播报** | **千元级** | **当前为固定点位/巡检原型** |

## 5. 推荐 Agent 分工

### 5.1 Product Scope Agent

职责：

- 维护 PRD v2.2 范围
- 把控 P0/P1/P2 边界
- 统一"无障碍辅助通行"口径
- 审核是否偏离校园最后 20 米主场景
- 管理提交材料的产品叙事（论文/PPT/展架/视频）
- 竞品对比表准确性审查

输出：

- 任务拆解
- 里程碑验收标准
- 演示场景定义
- PPT/论文/展架口径审查

禁止：

- 私自把项目改回"满街穿戴导盲"
- 为了炫技加入非 P0 功能
- 在主链路未稳时推动 App 或云端作为硬依赖
- 把"原型系统"包装成"成熟产品"

### 5.2 Edge CV Agent

职责：

- Jetson 环境维护
- 摄像头/图片/视频输入链路
- YOLO 推理（PyTorch -> ONNX -> TensorRT FP16）
- 自定义数据训练
- FPS、延迟记录
- 模型性能基准测试

关注指标：

- 推理帧率目标 `>=30 FPS`
- 单帧推理延迟目标 `<100 ms`
- 检测到播报响应目标 `<1.5 s`
- P0 场景识别成功率目标 `>=80%`
- 可连续演示运行，不临场崩溃

止损：

- CSI 相机未恢复时，先用 USB 摄像头、视频文件或图片序列打通闭环
- TensorRT 未完成时，先用 PyTorch/Ultralytics 演示
- 自定义模型未完成时，先用 COCO 代理路障演示链路
- 当前 COCO 代理映射必须诚实表述为"冷启动策略"

### 5.3 Edge Interaction Agent

职责：

- 事件聚合与去抖
- 模板化语义提示（<50字短句）
- 本地 mp3 播报
- `presenter_female` 音频缓存接入
- 播报冷却和优先级控制
- 多事件仲裁（全局音频锁）
- 断网降级策略（MiniMax -> pyttsx3 -> print）

必须遵守：

```text
默认音色：presenter_female
默认模型：speech-2.8-hd
现场播报优先使用本地缓存 mp3
启动时预加载全部模板缓存
```

输出：

- `AudioOutput` 后端
- 播报缓存说明
- 播报延迟测试
- 音频失败降级策略
- 信息密度控制（低信息密度原则）

### 5.4 Data & Evaluation Agent

职责：

- P0 数据采集（目标 >=600 张）
- 类别定义和标注规范
- 训练/验证/测试划分
- mAP、FPS、延迟、实地测试记录
- 论文实验表格

P0 数据类别与目标分布：

- `blind_road_occupied`：35%（~210 张）
- `stairs`：20%（~120 张）
- `ramp`：20%（~120 张）
- `road_obstacle`：25%（~150 张）

硬约束：

- 不新增训练主类 `traffic_light`
- 不把 P1 类别拖进当前 baseline
- 每批数据必须记录场景、日期、质量问题
- 测试集必须包含 >=20% 自采数据验证真实场景泛化性
- 同一地点/场景的图片不能分散在不同集合（防泄漏）

### 5.5 Integration & Demo Agent

职责：

- 校园最后 20 米演示路线（A-D 点位）
- 演示道具准备
- 端到端联调
- 录屏/视频/照片（4:30 成片）
- 现场 checklist

必须跑通的演示：

1. 盲道占用 -> 语音提示
2. 台阶 -> 减速提示
3. 坡道或路面障碍 -> 通行/避让提示

展示必须包含：

- 输入画面
- 检测框/类别/置信度
- 事件日志
- 语音播报
- Jetson 设备特写（证明边缘部署）

演示视频要求：

- 时长：<=5 分钟
- 分辨率：1920x1080
- 大小：<=100M
- 着装：正装或商务休闲+深色长裤/长裙
- 不得出现校名、人脸、车牌

### 5.6 Cloud & Mobile Agent（P1 阶段激活）

职责：

- 可选 FastAPI 接收结构化事件
- 可选 Flutter 展示页
- 可选大模型长文本报告生成（Kimi/Qwen）
- 可选主动语音查询

当前阶段规则：

- 不阻塞 P0
- 不作为现场唯一主链路
- 云端失败不能影响本地播报
- 大模型只做事后报告，不做实时推理（避免 3-5 秒延迟）
- App 没完成不影响比赛版提交

## 7. 统一数据与接口约定

### 7.1 事件字段

P0 事件最小字段：

- `event_id`
- `node_id`
- `timestamp`
- `type`
- `confidence`

推荐字段：

- `bbox`
- `distance_m`
- `direction`
- `source_frame_ids`
- `location`

原则：

- `distance_m` 和 `direction` 不稳定时可以为空
- 不得伪造成必填项
- 不上传原始图像或视频

### 7.2 事件类型

P0：

- `blind_road_occupied`
- `stairs`
- `ramp`
- `road_obstacle`

P1 预留（代码隔离，不可触发）：

- `traffic_red`
- `traffic_green`
- `crosswalk`

### 7.3 语义提示

必须满足：

- 中文
- 简短（<50 字）
- 明确
- 可执行
- 不含不确定措辞（"可能""也许""大概"）
- 同一障碍不重复播报（冷却时间内）
- **低信息密度**：宁可少报，不可多报（黄金原则）

推荐：

```text
前方盲道被占用，请注意绕行。
前方有台阶，请减速。
前方有坡道，位于右侧。
右前方有障碍，请注意避让。
```

### 7.4 音频系统规范

- 默认音色：`presenter_female`
- 默认模型：`speech-2.8-hd`
- 缓存目录：`~/.cache/linkable_edge/tts_minimax/`
- 缓存键：SHA256(文本+音色+速度+情绪+模型)
- 启动时必须预加载全部模板缓存
- 降级链：MiniMax -> pyttsx3 -> print
- 失败时绝不静默

## 8. 风险响应

### 8.1 CSI 相机不出帧

判据：

- `/dev/video0` 存在但 JPEG 为 0 字节
- Argus / V4L2 出现 timeout

动作：

1. 换排线
2. 换 CSI 口
3. 换摄像头
4. 切 USB 摄像头
5. 使用视频文件/图片序列演示闭环

不得继续把项目卡死在 CSI 单点问题上。

### 8.2 模型训练来不及

动作：

- 保留 COCO 代理类演示路障
- 先完成链路闭环
- 数据和自定义模型作为逐步增强
- **诚实表述**："当前为 COCO 代理冷启动阶段，自定义模型训练是下一步"

### 8.3 音频不稳定

动作：

- 使用本地缓存 mp3
- 使用 `mpg123`
- 播报失败时显示文本（print 兜底）
- 启动时预加载全部缓存
- 现场准备小音箱和备用线材

### 8.4 演示现场网络不稳定

动作：

- 不调用云端 TTS
- 不依赖云端 LLM
- 不依赖 Flutter 主流程
- 完全走本地模板和缓存音频

### 8.5 自采数据不足

动作：

- 即使只有 20 张/类，也要拍，证明"数据采集已启动"
- 重点补齐 blind_road_occupied（演示核心）
- 公共数据集作为 cold-start 补充
- 同一地点多时段、多角度采集

基于边缘智能的无障碍辅助通行原型系统

## 10. Review Checklist

每次重要改动后至少检查：

### 产品范围

- [ ] 是否仍以 P0 为主线
