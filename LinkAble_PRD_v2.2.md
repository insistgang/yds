## 0. 一句话定义

**LinkAble 是一套基于边缘智能的无障碍辅助通行感知系统，部署于人行无障碍空间的关键节点，实时识别盲道占用、台阶、坡道、路面障碍等通行风险，通过语音播报为视障与行动不便人群提供即时通行提示，同时沉淀结构化事件数据支撑无障碍环境治理。**

一句话对比：

> 无人驾驶让车看懂路，LinkAble 让弱势行人听懂路、让管理者看见盲区。

## 1. 项目本质与边界（必读）

### 1.1 这是什么

LinkAble 是一个**边缘智能感知原型系统**，不是一款穿戴式导盲产品。它的技术内核是：

```
视觉感知 → 事件理解 → 本地语音提示 → 结构化记录
```

### 1.2 这不是什么

- ❌ 不是成熟的穿戴式导盲硬件
- ❌ 不能替代盲杖、导盲犬或用户自主判断
- ❌ 不直接控制用户行动，不承担安全责任
- ❌ 不是手机 App，不是云平台，不是志愿者系统

### 1.3 为什么这样定位

Jetson Orin Nano 需要持续供电与散热，长期穿戴在视障用户身上不现实，且开放道路场景的安全责任边界过重。本项目的合理形态是**部署在固定节点或用于巡检的边缘感知设备**，而非穿戴式产品。

穿戴形态仅作为**演示形态**保留，用于拍摄"用户靠近障碍并听到提示"的视频片段，不构成产品主线。

## 2. 应用价值

### 2.1 让无障碍环境的隐性问题变可见（核心）

盲道被共享单车占用、坡道入口堆放杂物、路面临时障碍——这些问题在城市管理者视角是隐性的，缺乏系统化的数据支撑。LinkAble 通过结构化事件记录（时间、位置、类型、置信度），把这些隐性问题转化为可量化、可追溯、可治理的数据。

**这是项目最有可能真正落地的价值。**

### 2.2 填补"最后几米"的辅助提示空白

现有导航 App 解决"从 A 走到 B"，盲杖只能探测 1 米内有无障碍物，均无法回答"眼前这一步是什么、怎么应对"。LinkAble 在固定节点提供局部通行风险的实时提示，填补这一空白。

### 2.3 低成本可部署的边缘智能骨架

单摄像头 + Jetson 边缘推理，无需改造基础设施。技术骨架可迁移至工厂巡检、园区安防、设施监测等需要"实时识别 + 即时响应"的场景。

## 5. 系统架构

### 5.1 主链路（边缘端，实时）

```
摄像头/图片/视频输入
  → YOLO 检测（yolo11n.pt）
  → 事件聚合与去抖（EventBuilder）
  → 模板化中文提示（semantics.py）
  → 本地音频播报（audio.py / MiniMax TTS）
  → 结构化事件记录（publisher.py）
```

### 5.2 增强链路（云端，离线，P1）

```
结构化事件流（JSON）
  → 大模型分析（Kimi/Qwen）
  → 周/月度无障碍报告
  → 治理建议输出
```

> 大模型仅用于离线分析报告生成，**不进入实时避障链路**，避免引入网络延迟和不确定性。

### 5.3 设计原则

- **端侧实时优先**：实时避障链路全部在边缘完成，<1.5 秒响应
- **本地音频缓存**：演示现场不依赖云端 TTS
- **大模型离线增强**：云端用于事后数据分析，不影响实时性
- **规则系统兜底**：所有关键决策由确定性规则做出，LLM 不参与安全相关决策

### 5.4 推荐模块

| 模块 | 职责 |
|---|---|
| `detector.py` | YOLOv11 推理，P0 类别过滤 |
| `event_builder.py` | 连续帧去抖、冷却控制、优先级排序 |
| `semantics.py` | 事件转模板化中文提示 |
| `audio.py` | 本地 mp3 播报、失败降级 |
| `pipeline.py` | 串联完整主链路 |
| `publisher.py` | 可选 HTTP 上报结构化事件 |
| `usb_demo.py` | 主演示入口 |
| `report_gen.py`（P1） | 离线事件流 → LLM → 报告 |

### 5.5 部署形态（已重新排序）

| 形态 | 当前定位 | 应用场景 |
|---|---|---|
| **固定点位部署** | **推荐主形态** | 盲道入口、坡道入口、楼宇门口、地铁口 |
| **手持/推车式巡检** | 推荐主形态 | 校园、社区无障碍环境周期性巡检 |
| 短距离佩戴 | **演示形态** | 仅用于拍摄演示视频片段 |

> 形态排序对齐项目真实价值：固定部署和巡检是落地形态，穿戴只是演示道具。

### 6.1 P0 必做

- 红绿灯识别、人行横道识别

- 实时大模型语义增强（与实时性矛盾）
- 用户主动语音问答（延迟链路过长，演示风险大）

## 8. 语音播报策略

### 默认音色

```
model = speech-2.8-hd
voice_id = presenter_female
```

### 播报原则

- 句子短，优先 <50 字
- 必须包含风险类型
- 有条件时包含距离或方向
- 避免"可能、似乎、也许"等不确定措辞
- 不连续重复播报同一障碍
- **信息密度低、决策清晰**——视障辅助的黄金原则

### 推荐文案

```
前方盲道被占用，请注意绕行。
前方有台阶，请减速。
右前方有障碍，请注意避让。
前方有坡道，位于右侧。
```

### 降级策略

| 问题 | 降级 |
|---|---|
| MiniMax 网络不可用 | 使用本地缓存 mp3 |
| mp3 缺失 | 终端文字输出 |
| 音频设备异常 | 显示字幕/日志 |
| 同一事件重复触发 | 冷却时间内不播报 |

## 15. 对外口径

### 30 秒介绍

LinkAble 是一套基于边缘智能的无障碍辅助通行感知系统。系统通过 Jetson 和摄像头部署在人行无障碍空间的关键节点，实时感知盲道占用、台阶、坡道和路面障碍，在端侧完成事件聚合和本地语音播报。相比自动驾驶关注车辆行驶，LinkAble 关注弱势行人眼前几米的安全通行；相比传统盲杖只能物理探测，LinkAble 能识别障碍类型并语义化提示；相比 App 只能宏观导航，LinkAble 能感知局部环境并即时反馈。同时，结构化事件数据可沉淀为无障碍环境治理的量化依据。

### 展架主标题

```
从"车能自己开"到"人能安全走"
```

### 展架副标题

```
基于边缘智能的无障碍辅助通行感知系统
```

### 核心卖点

- 面向人行无障碍空间，而非机动车道
- 单摄像头 + Jetson 边缘推理，低成本可部署
- 检测结果转语义化中文播报，用户可直接执行
- 结构化事件数据沉淀为无障碍治理依据
- 边缘实时 + 云端离线大模型分析的双链路架构

### 应对常见质疑的标准答案

**Q：怎么戴在盲人身上？**
> A：当前版本是边缘智能原型系统，主要部署形态是固定节点和手持巡检，不是穿戴式产品。穿戴形态仅作为演示场景。穿戴化是后续轻量化硬件设计的工作方向。

**Q：和盲杖/导盲犬比有什么优势？**
> A：盲杖只能物理探测 1 米内的存在，不识别类型；导盲犬成本高、稀缺；LinkAble 识别障碍类别并语义化播报，且数据可沉淀为治理依据。三者不是替代关系而是互补。

**Q：为什么不直接用大模型？**
> A：实时避障链路对延迟敏感（<1.5秒），大模型推理延迟（3-5秒）会破坏实时性。我们将大模型用于离线事件流分析与无障碍报告生成，发挥其语义理解能力，而不引入实时性风险。

### 禁止说的

- "已经做出了成熟导盲设备"
- "可以替代盲杖或导盲犬"
- "可以保证视障用户完全安全"
- "核心是 App/云平台/志愿者系统"
