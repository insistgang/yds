## 1. 系统总览

### 1.1 一句话定义

LinkAble 是一套基于边缘智能的无障碍辅助通行感知系统，部署于人行无障碍空间的关键节点，通过 YOLO 视觉识别盲道占用、台阶、坡道、路面障碍，端侧实时语音播报，并沉淀结构化事件数据支撑无障碍治理。

### 1.2 主链路（边缘端，实时）

```
摄像头/图片/视频输入 (inputs.py)
  → YOLOv11 检测 (detector.py)
  → 事件聚合与去抖 (event_builder.py)
  → 模板化中文提示 (semantics.py)
  → Pipeline 仲裁（一帧只播最高优先级）
  → 本地音频播报 (audio_manager.py)
  → 结构化事件记录 (publisher.py)
  → 性能基准 (benchmark.py)
```

### 1.3 增强链路（云端，离线，P1）

```
结构化事件流 (JSON) → 大模型分析 (Kimi/Qwen) → 周/月度无障碍报告
```

> 大模型**仅用于离线分析**，不进入实时避障链路。规则系统负责安全决策，大模型负责事后洞察。

### 1.4 P0 检测四类与优先级

| 类别 | 英文名 | 优先级 | 说明 |
|---|---|---|---|
| 台阶 | `stairs` | 100 | 安全风险最高 |
| 路面障碍 | `road_obstacle` | 80 | 锥桶、箱子、围挡 |
| 盲道占用 | `blind_road_occupied` | 70 | 盲道被占用 |
| 坡道 | `ramp` | 60 | 无障碍坡道 |

### 1.5 不可违背的工程约束

- **本地优先，云端增强**：能本地播报就不依赖云端；能用缓存 mp3 就不现场请求 TTS。
- **离线鲁棒**：全网断连下系统仍能 100% 确定性响应。
- **隐私硬约束**：不存原始视频、不上传连续图像流、仅上报结构化事件、位置匿名化。
- **规则兜底**：所有安全相关决策由确定性规则做出，LLM 不参与。

## 3. 第 1 层：输入层 `inputs.py`

**职责**：多源输入适配，统一读取接口，异常自动切换。

### 核对项

- [ ] 支持 4 种输入源：CSI 相机(IMX219) / USB 相机(/dev/video0) / 视频文件(.mp4) / 图片序列(.jpg/.png)。
- [ ] `auto_select_source()` 存在，回退优先级为 **CSI 故障 → USB → 视频 → 图片**。
- [ ] 统一读取接口：所有源返回同一种帧格式，下游无需区分来源。
- [ ] 异常自动切换：当前源失败时自动降级到下一级，不崩溃。

### 补缺动作

- 补 `auto_select_source()` 回退链，每级失败要有日志，不能静默。
- 补图片序列 / 视频文件回退路径，确保**无相机环境也能跑**（离线演示命脉）。

### 验收证据

- [ ] 无 CSI 相机环境运行，确认自动落到 USB/视频/图片，打印实际选中的源。

## 4. 第 2 层：检测层 `detector.py`

**职责**：YOLOv11 推理，P0 四类过滤，COCO 冷启动代理，异常降级，性能基准。

### 核对项

- [ ] YOLOv11 (PyTorch) 推理跑通。
- [ ] P0 四类正确定义并过滤：`blind_road_occupied` / `stairs` / `ramp` / `road_obstacle`。
- [ ] COCO 冷启动代理映射存在：`bicycle/bench/person → road_obstacle`，且 `stairs/ramp/blind_road` 明确标注"需自定义训练"。
- [ ] 配置项正确：`conf >= 0.25`，`imgsz=640`。
- [ ] 后端字段预留：`pytorch / onnx / tensorrt`（engine 字段）。
- [ ] 异常降级：单帧推理失败返回空结果而非崩溃。
- [ ] 性能基准：`benchmark.py` 能记录 pre / infer / post 三段耗时。

### 补缺动作

- COCO 代理映射若无"诚实标注"，补一个明确区分"代理类"与"自训练类"的配置——**答辩靠它讲冷启动边界**。
- `tensorrt` 若仅预留字段未实现：**保持预留即可，冲刺期不实装 TensorRT**（PyTorch 原生够用，实装风险高）。
- 单帧失败降级若缺，补 try/except 返回空 `FrameDetections`。

### 验收证据

- [ ] 跑一张测试图，打印检测框、类别、置信度。
- [ ] 故意喂坏图，确认返回空结果且不崩。

## 5. 第 3 层：事件层 `event_builder.py`

> **重点核对层**——直接关系到 EventBuilder 消融实验的可行性。

**职责**：连续帧去抖、冷却、内存清理、优先级仲裁、P1 硬隔离。

### 核对项

- [ ] 连续帧检测：`min_consecutive_frames = 2`（同一目标连续 N 帧才确认）。
- [ ] 发射冷却：`emit_cooldown_frames = 5`（同类事件冷却期内不重复发射）。
- [ ] 内存清理：TTL 过期自动删除 `_states`，防止内存泄漏。
- [ ] 优先级排序为 **stairs(100) > road_obstacle(80) > blind_road(70) > ramp(60)**。
- [ ] P1 硬隔离：`traffic_red / crosswalk` 被直接过滤，不进入 P0 链路。
- [ ] 输出 `DetectionEvent` 含全部字段：`event_id / label / confidence / timestamp / priority / bbox / distance_m / direction / source_frame_ids`。

### 补缺动作

- **关键：确认去抖与冷却两个参数是可配置开关**（消融实验靠开关它们产出对比数据）。若写死为常量，改成可从配置 / 参数注入。
- `source_frame_ids` 若没记录，补上——它是证明"多帧聚合"的证据字段。

### 验收证据

- [ ] 喂一段连续帧，确认抖动被 `min_consecutive_frames` 滤掉。
- [ ] 确认冷却期内同类事件不重复输出。
- [ ] **为消融实验预留入口**：能通过参数关闭去抖/冷却，跑出"未去抖"基线。

## 6. 第 4 层：语义层 `semantics.py`

**职责**：事件转模板化中文提示，方向 / 距离信息，模板预加载。

### 核对项

- [ ] 模板化中文提示均 <50 字、短句、可执行（如"前方盲道被占用，请注意绕行"）。
- [ ] 方向信息支持：前方 / 左前方 / 右前方 / 左侧 / 右侧。
- [ ] 距离信息支持"前方 X 米"，且**不稳定时省略距离**（避免给错误距离）。
- [ ] `ALL_TEMPLATE_TEXTS` 覆盖全部 28 条模板，启动时预加载。
- [ ] 模板文本与 TTS 缓存一一对应（每条模板都有预生成音频）。

### 补缺动作

- 距离"不稳定时省略"逻辑若无，补置信度 / 距离估计阈值，低于阈值不播距离。
- 模板数量与 TTS 缓存数量若不一致，补齐到一致（否则现场触发在线合成，破坏 5ms 延迟）。

### 验收证据

- [ ] 对四类事件各生成一句提示，确认 <50 字、含方向、距离按规则省略。

## 7. 第 5 层：Pipeline 仲裁层

**职责**：一帧只播最高优先级事件，其余丢弃或排队。

### 核对项

- [ ] 实现"一帧只播最高优先级事件"（多事件冲突时只取 priority 最高的）。
- [ ] 其余事件丢弃或排队，行为明确、有日志。
- [ ] 串联完整主链路（input → detect → event → semantic → audio → publish）。

### 补缺动作

- 仲裁逻辑若分散在各处，集中到 pipeline 一处，便于答辩指认"清晰播报"的实现位置。

### 验收证据

- [ ] 构造一帧同时含 stairs + ramp 的场景，确认只播 stairs（高优先级）。

## 8. 第 6 层：音频层 `audio_manager.py`

> **重点核对层**——直接关系到"5ms 延迟"卖点和"高频被问点"。

**职责**：本地播报、全局音频锁、启动预加载、三重降级。

### 核对项

- [ ] 音色为 `presenter_female`（女声），TTS 模型 `speech-2.8-hd`(MiniMax)。
- [ ] **全局音频锁 `threading.Lock`** 实装（防止多事件同时播报互相打断）。
- [ ] **启动预加载**：启动时批量合成全部模板音频并缓存。
- [ ] 降级链为 **MiniMax(API) → pyttsx3 → print 输出**（从高到低三级）。
- [ ] 实测"缓存命中时播报启动延迟 < 5ms"（对比在线合成 ~700ms）。

### 补缺动作

- **全局音频锁是高频被问点**（评委会问"两个障碍同时出现怎么办"）。若缺，必须补。
- 降级链若不完整，补到三级——断网 / 无音频设备时仍能 `print` 兜底，这是离线鲁棒性的证据。

### 验收证据

- [ ] 触发两个事件，确认音频锁串行播报、不重叠。
- [ ] 断网，确认降级到 pyttsx3 或 print，不崩。
- [ ] 测一次缓存命中延迟，记录实测数字（回填论文的"5ms"）。

## 9. 第 7 层：上报层 `publisher.py`

**职责**：结构化事件上报，断网不崩，隐私合规。

### 核对项

- [ ] `SafePublisher` 实现：断网不崩 + 失败重试 + 本地缓存。
- [ ] **只上传结构化事件数据，不上传原始图像**（隐私硬约束）。
- [ ] 接口为 `POST /api/v1/detect`，`Content-Type: application/json`。

### 补缺动作

- 上报若因断网阻塞主链路，改成异步 / 非阻塞，**绝不能让上报拖垮实时播报**。
- 确认本地缓存落盘，网络恢复后可补传。

### 验收证据

- [ ] 断网状态下触发事件，确认主链路照常播报、事件落本地缓存、无异常。

## 10. 第 8 层：性能基准层 `benchmark.py`

**职责**：实时指标 + 阶段耗时 + 结构化输出。

### 核对项

- [ ] 输出实时指标：FPS、延迟(ms)、吞吐量。
- [ ] 分段记录：pre(预处理) / infer(推理) / post(后处理)。
- [ ] 输出 JSON 详情 + CSV 汇总（用于回归与对比分析）。
- [ ] 对照目标基准：FPS >= 30、单帧延迟 < 100ms。

### 补缺动作

- 若无 CSV 汇总，补上——消融实验和指标对比都靠它出表。
- 补"端到端延迟"测量（检测到播报，目标 <1.5s），这是架构核心卖点的数字。

### 验收证据

- [ ] 跑一轮基准，产出 JSON + CSV，确认含 FPS / 单帧延迟 / 端到端延迟三个数。

## 11. 关键证据产出（四件生死活）

> 前 8 层验证系统完整性。这一部分产出答辩真正缺的证据，**优先级高于一切卫生级清理**。

### 11.1 训练 baseline，跑出真实指标 🔴

- [ ] test 集上输出 mAP50 / Recall。
- [ ] Jetson 实测单帧推理延迟、FPS。
- [ ] 端到端掐表：检测到播报延迟（目标 <1.5s）。
- [ ] 对比 COCO 代理 vs 自训练模型，诚实标注冷启动边界。

> 要的是**真实数字**，不是漂亮数字。mAP 75% + 诚实说明 > 编的 90%。评委专门戳编的数。

### 11.2 EventBuilder 消融实验 🔴

- [ ] 关闭去抖 + 冷却跑一遍，记录"单位时间播报次数"。
- [ ] 打开去抖 + 冷却跑一遍，记录"单位时间播报次数"。
- [ ] 输出对比表 / 图：证明从"1 秒播 30 次"到"1 次清晰播报"。

> 这把论文的 EventBuilder 创新点从自夸变成实验证据。投入半天，回报一张能进论文的图表。

### 11.3 固定点位演示录制 🟡

- [ ] 跑通离线演示：检测框 + 事件日志 + 语音播报四件套同屏。
- [ ] **B 点（路面障碍）必须稳**，A/C/D 尽力。
- [ ] 记录连续运行时长、现场延迟。

> 战术：一个稳的点位 > 四个翻车的点位。
